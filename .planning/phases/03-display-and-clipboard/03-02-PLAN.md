<objective>
Implement clipboard reading (local side) and clipboard writing (remote side) in a single clipboard module.

Purpose: The local daemon needs to watch the system clipboard for new screenshot images and encode them as PNG for transmission. The remote daemon needs to write received PNG images to the system clipboard using the correct tool for the detected display environment. This is the core clipboard bridge logic.

Output: `src/clipboard.rs` with `watch_clipboard()` for the local side and `ClipboardWriter` for the remote side.
</objective>

  <name>Task 1: Implement ClipboardReader (local side watch_clipboard)</name>
  <files>src/clipboard.rs</files>
  <action>
Replace the stub in `src/clipboard.rs` with clipboard reading implementation. This task adds the local-side watcher; Task 2 adds the remote-side writer to the same file.

**Public API:**
- `pub async fn watch_clipboard(tx: tokio::sync::mpsc::Sender<Vec<u8>>) -> anyhow::Result<()>` — runs forever, sends PNG bytes when clipboard changes

**Implementation:**

1. **Display auto-detection (CLRD-04):** At the start of `watch_clipboard`, detect whether we're on Wayland or X11:
   - Check `$WAYLAND_DISPLAY` — if set and non-empty, log "clipboard: using Wayland" at info level
   - Otherwise check `$DISPLAY` — if set and non-empty, log "clipboard: using X11" at info level
   - If neither, return `Err(anyhow!("No display server found. Set WAYLAND_DISPLAY or DISPLAY."))` with clear error

2. **arboard initialization:** Create `arboard::Clipboard::new()` inside `tokio::task::spawn_blocking` (arboard's X11 backend is `!Send` and does blocking I/O). All arboard calls must happen inside spawn_blocking blocks.

3. **Startup-skip (per locked decision):** On first iteration, call `get_image()`. If it succeeds, compute the SHA-256 hash of the raw RGBA bytes and store as `last_hash`. Do NOT send this initial image. This prevents re-sending a stale screenshot from a previous session.

4. **Polling loop (CLRD-01):** Every 500ms:
   - Call `clipboard.get_image()` inside spawn_blocking
   - On success: compute SHA-256 of RGBA bytes using `sha2::Sha256::digest()`
   - If hash differs from `last_hash`: encode RGBA to PNG using `image::RgbaImage::from_raw(width, height, bytes.to_vec())` then `img.write_to(&mut Cursor::new(&mut buf), ImageFormat::Png)`, send PNG bytes over `tx`, update `last_hash`, log at info: "clipboard image captured ({kb} KB), sending"
   - If hash matches: skip (deduplicated), log at debug level
   - On `Err` from get_image: no image on clipboard, log at debug level only

5. **RGBA-to-PNG helper:** `fn rgba_to_png(width: u32, height: u32, rgba: &[u8]) -> anyhow::Result<Vec<u8>>` using the image crate pattern from research. Return error if dimensions don't match byte count.

6. **Content hash helper:** `fn content_hash(bytes: &[u8]) -> [u8; 32]` using `Sha256::digest(bytes).into()`

**Wayland vs X11 (CLRD-02, CLRD-03):** arboard with the `wayland-data-control` feature handles both transparently. The same `Clipboard::new()` and `get_image()` calls work on both. No separate code paths needed for reading — arboard auto-detects. The display detection at the top is for logging only.

**Error on missing tools:** arboard handles this internally — if the X11 or Wayland backend can't connect, `Clipboard::new()` returns an error. Catch this error and provide a clear message: "Failed to access clipboard. Ensure a display server is running." On Wayland, if `wl-paste` is needed as a fallback, arboard falls back automatically.

**Logging:** Use `tracing::info!` for image captures. Use `tracing::debug!` for per-poll details (no image, hash match). Include timestamps (handled by tracing-subscriber already configured in main.rs).

<task type="auto">
  <name>Task 2: Implement ClipboardWriter (remote side) with subprocess dispatch</name>
  <files>src/clipboard.rs</files>
  <action>
Add the ClipboardWriter to `src/clipboard.rs` (same file as Task 1).

**Public API:**
- `pub struct ClipboardWriter` with private fields: `current_child: Option<tokio::process::Child>`, `display: DisplayEnvironment`
- `pub fn new(display: DisplayEnvironment) -> Self`
- `pub async fn write(&mut self, png_bytes: &[u8]) -> anyhow::Result<()>`

**Tool availability check:**
- `pub async fn check_clipboard_tools(display: &DisplayEnvironment) -> anyhow::Result<()>` — called at daemon startup before constructing ClipboardWriter
- For `DisplayEnvironment::X11` or `DisplayEnvironment::Xvfb`: check `which xclip` — if not found, bail with "xclip not found. Install with: sudo apt install xclip"
- For `DisplayEnvironment::Wayland`: check `which wl-copy` — if not found, bail with "wl-copy not found. Install with: sudo apt install wl-clipboard"
- For `DisplayEnvironment::Headless`: bail with "No display available for clipboard write. Run display detection first."

**write() implementation (CLWR-01 through CLWR-05):**

1. **Kill previous child (CLWR-05 support):** If `self.current_child` is Some, call `child.kill().await` to stop the previous clipboard-holding process. This prevents zombie accumulation. Log at debug level: "killing previous clipboard holder"

2. **Dispatch by display environment (CLWR-04):**
   - `DisplayEnvironment::Wayland` (CLWR-03): Spawn `wl-copy --type image/png` with stdin piped
   - `DisplayEnvironment::X11` (CLWR-02): Spawn `xclip -selection clipboard -t image/png -i` with stdin piped. Pass `-display` flag using current `$DISPLAY` value.
   - `DisplayEnvironment::Xvfb` (CLWR-02): Same as X11 but use the display string from DisplayManager (passed via `$DISPLAY` env var already set by display.rs)
   - `DisplayEnvironment::Headless`: Return `Err(anyhow!("no display available for clipboard write"))`

3. **Pipe PNG data:** Take stdin from child, write all png_bytes, then drop stdin handle (signals EOF). xclip and wl-copy read all data then fork to background to serve SelectionRequest events.

4. **Store child handle (CLWR-05):** Set `self.current_child = Some(child)`. Do NOT call `.wait()` — the subprocess must stay alive to serve clipboard paste requests. This is critical for X11 selection ownership persistence.

5. **Logging:** `tracing::info!("clipboard write: {} KB", png_bytes.len() / 1024)`

**CRITICAL anti-patterns to avoid:**
- Do NOT set `kill_on_drop(true)` on the Command — would kill the clipboard holder when the Child handle is dropped during reassignment
- Do NOT call `child.wait()` after piping data — xclip/wl-copy need to stay alive
- Do NOT use arboard `set_image()` for writes — drops clipboard when Clipboard instance is dropped

**Import requirements:** `use crate::protocol::DisplayEnvironment;`, `use tokio::process::Command;`, `use tokio::io::AsyncWriteExt;`
